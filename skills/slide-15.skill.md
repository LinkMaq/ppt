# Slide 15: 全景可观测性与动态监控

## 页面逻辑与数据映射
- **主题**：用一张全景监控墙解释 Rune 如何把集群状态、资源利用、API 调用链与审计事件统一收口。
- **核心观点**：企业级可观测性不只是“看监控图”，而是让异常、资源浪费与变更责任在同一页面内被同时看见。
- **本页约束理解**：
  - 延续全局白底、浅蓝网格、柔光渐变和企业科技风。
  - 不复用既有母版，不沿着前页版式继续推导，而是单独设计“运营监控墙”结构。
- **信息映射**：
  - 左侧告警矩阵：承接高优先级异常、路由抖动和发布变更，先建立“问题入口”。
  - 中央主舞台：承接监控面板、资源使用分析和关键 KPI，是页面第一视觉。
  - 右侧时间线：承接 API、实例和配置变更的追踪，补足“谁做了什么”的审计视角。
  - 底部结论条：把“可观测性从被动排查变成主动定位”收束成一句话。

## 布局体系 (Operations Wall)
采用 **顶部标题 + 左中右监控墙 + 底部结论条**：
1. **顶部标题区**
   - 标题直接点明“全景可观测性与动态监控”。
   - 副标题强调节点、工作负载、调用链和变更行为统一可见。
2. **摘要条**
   - 五个标签提前交代本页能力边界：大盘、资源双视角、审计追踪、变更回放、告警收口。
3. **左侧告警矩阵**
   - 三张告警卡覆盖严重异常、预警和审计事件。
   - 底部补一组服务健康状态，增强“持续监控”感。
4. **中央主舞台**
   - 顶部 KPI 先给出监控规模感。
   - 中部拆成两块：左边是节点与工作负载全景，右边是资源分配率与实际使用率分析。
   - 底部三张追踪卡承接 API 审计、实例生命周期和配置变更回放。
5. **右侧审计时间线**
   - 以时间顺序承接 fallback、灰度发布、RBAC 策略变更和告警恢复。
   - 最下方一张覆盖率卡，说明关键动作全部留痕。

## 视觉风格
- **整体方向**：白底、冷色渐变、轻阴影，但视觉语义从“产品页”切到“运营页”。
- **背景**：
  - 使用浅网格和大面积柔光维持秩序感。
  - 中部横向光带和大圆环只做弱感知，暗示持续的数据流动。
- **组件语言**：
  - 左右两侧采用轻量白卡，突出监控信息密度。
  - 中央主舞台作为页面重心，使用更厚的卡片层级和图表区域。
  - 图表不追求真实截图，而是通过 SVG 曲线、双层进度条和时间线表达“监控联动”。
- **品牌映射**：
  - 蓝色：平台指标、调用链、基础监控
  - 绿色：健康状态、恢复、稳定运行
  - 紫色：配置漂移、变更回放、复杂治理信号

## 页面目标
- 让观众快速理解 Rune 的可观测性覆盖面，不只是基础设施监控，还包含业务调用与治理审计。
- 让平台团队的运营视角具象化：异常、资源和责任链都在一个界面内闭环。
